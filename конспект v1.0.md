# Конспект по основам тестирования ПО

## Основные понятия

**Верификация** - проверяем, что продукт создаётся ПО ПРАВИЛАМ. Пример: проверка ТЗ, ревью кода.

**Валидация** - проверяем, что продукт НУЖЕН пользователям. Пример: тестирование готовой функции.

**Тестирование** - практическая часть валидации. Ищем баги, выполняя программу.

**QA (Quality Assurance)** - более широкое понятие, чем тестирование. Управление ВСЕМИ процессами для качества.

## Процесс тестирования (шаги)

1. **Анализ требований** - читаем документы, ищем проблемы
2. **Планирование** - решаем, что, как и когда тестировать
3. **Проектирование тестов** - создаём тест-кейсы (инструкции для проверки)
4. **Подготовка** - настраиваем среду для тестов, готовим данные
5. **Выполнение** - запускаем тесты, находим баги, записываем
6. **Отчётность** - фиксируем результаты, сообщаем команде
7. **Завершение** - проверяем исправления, делаем итоговый отчёт

## Кто занимается тестированием?

- **Manual QA** - тестирует вручную, как пользователь
- **Automation QA** - пишет программы для автоматических тестов
- **Performance QA** - проверяет скорость и нагрузку
- **QA Lead** - руководит командой
- **SDET/QE** - универсальный специалист (разработка + тестирование)

## Что нужно знать новичку?

### Основные виды тестирования:
- **Функциональное** - работает ли функция?
- **Нефункциональное** - насколько хорошо работает? (скорость, безопасность)
- **Дымовое (Smoke)** - быстрая проверка главных функций
- **Регрессионное** - проверяем, не сломали ли старое при добавлении нового
- **Исследовательское** - свободное тестирование без готовых сценариев

### Уровни тестирования:
1. **Модульное (Unit)** - проверка маленьких частей кода (задача разработчиков)
2. **Интеграционное** - проверка взаимодействия частей
3. **Системное** - проверка всей программы
4. **Приемочное (UAT)** - финальная проверка заказчиком

### Техники для создания тестов:
- **Классы эквивалентности** - группируем похожие данные
- **Граничные значения** - проверяем крайние случаи
- **Таблицы решений** - систематизируем условия и действия

## Инструменты (базовый набор)

- **Для хранения тестов**: TestRail, Qase
- **Для багов**: Jira, YouTrack
- **Для автоматизации UI**: Selenium, Playwright
- **Для API-тестов**: Postman
- **Для нагрузочного тестирования**: JMeter

## Современные подходы

**Agile/Scrum** - тестируем в каждом спринте (коротком цикле)

**Shift-Left** - тестируем как можно раньше

**CI/CD** - автоматические тесты запускаются при каждом изменении кода

## Важно запомнить

1. Тестирование не доказывает отсутствие ошибок
2. Цель - не просто найти баги, а улучшить продукт
3. Тестировщик должен мыслить критически
4. Коммуникация с командой так же важна, как технические навыки
5. Автоматизация экономит время на повторяющихся проверках

## Простые ответы на вопросы новичка

**В чем разница между тестировщиком и QA?**
Тестировщик ищет баги. QA-инженер ещё и улучшает процессы, чтобы багов было меньше.

**С чего начать карьеру?**
С ручного тестирования (Manual QA). Учитесь анализировать требования, писать тест-кейсы, находить и описывать баги.

**Что такое "тест-кейс"?**
Пошаговая инструкция для проверки функции. Пример: "1. Открыть сайт 2. Ввести логин 3. Ввести пароль 4. Нажать войти 5. Проверить, что зашли в систему".

**Что такое "баг"?**
Ошибка в программе. То, что работает не так, как должно.

**Нужно ли уметь программировать?**
Для ручного тестирования - не обязательно, но полезно. Для автоматизации - обязательно.

**Главный навык тестировщика?**
Умение думать, как пользователь, и находить проблемы ДО того, как их увидит пользователь.
